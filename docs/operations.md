# Operations

## Tuning performance

- Narrow the year range for faster ETL runs.
- Keep watchlist specific to avoid excessive matches.

## GitHub API rate limits

- Use `GITHUB_TOKEN` in CI.
- For local runs, optionally export `GITHUB_TOKEN`.

## Verify

- Run ETL twice and confirm consistent output

## Data Retention & Cache Management

### What's in `data/`

| File | Purpose | Safe to delete? | Regenerated by |
|---|---|---|---|
| `radar_data.json` | Current vulnerability data | Yes | `python etl.py` |
| `radar_report.md` | Human-readable report | Yes | `python etl.py` |
| `state.json` | Notification dedup state | Yes* | `python notify.py` |

\*Deleting `state.json` triggers a full baseline notification on the next run
(all current CVEs treated as new). This is safe but noisy.

### What's in `.nvd_cache/`

NVD feed JSON files, cached daily. These can grow to **~200 MB** over time.

**To prune:**

```bash
rm -rf .nvd_cache    # Fully re-downloaded on next ETL run
```

Or let the GitHub Actions cache handle it — the workflow uses date-based cache
keys that naturally expire.

### Recommended maintenance

For long-running forks, periodically:

1. Check `data/state.json` size — if it exceeds 5 MB, the CVE history is very
   large. The state auto-prunes entries older than 90 days on each run.
2. Clear `.nvd_cache/` if disk space is a concern.
3. `data/radar_data.json` is overwritten each run — it doesn't grow.
